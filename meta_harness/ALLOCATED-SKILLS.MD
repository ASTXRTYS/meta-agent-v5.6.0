## Agent-by-Agent Rationale

### Project Manager (PM)

The PM is the document hub. It assembles PRDs, evaluation criteria, stakeholder packages, and approval documents. `internal-comms` establishes a standard for how the PM communicates with both the stakeholder and the team ŌĆö status reports, decision memos, kickoffs, incident reports. `doc-coauthoring` provides the structured co-authoring workflow for all of the above. `prompt-architect` is a custom skill (to be created) that gives the PM prompt engineering capabilities ŌĆö the PM needs to be able to craft and refine system prompts for downstream agents as part of the scoping phase.

**Binary-format rendering (`.docx`/`.pdf`/`.pptx`) is delegated, not PM-native.** The PM does not carry the `docx`, `pdf`, or `pptx` skills directly. Rendering is owned by a single role-owned ephemeral Deep Agents SDK `SubAgent` dict spec named `document-renderer`, invoked through the built-in `task` tool (auto-installed by `SubAgentMiddleware`; see `.reference/libs/deepagents/deepagents/middleware/subagents.py:478`). The PM authors, coauthors, and curates the package content; only the terminal render-to-binary step is delegated. Rationale, SDK grounding, and wiring live in [`docs/specs/pm-document-renderer-subagent.md`](./docs/specs/pm-document-renderer-subagent.md). AD decision: `AD.md ┬¦4 Agent Primitive Decisions ŌåÆ PM-owned Document-Renderer SubAgent` (Q17).

### PM ŌåÆ `document-renderer` SubAgent

A stateless, ephemeral SDK `SubAgent` dict spec (the TypedDict at `.reference/libs/deepagents/deepagents/middleware/subagents.py:22-88`) owned by the PM. Not a peer Deep Agent, not mounted by the PCG, no checkpoint namespace, no handoff tools. It carries `docx`, `pdf`, and `pptx` as its `skills=[...]` list; `create_deep_agent()` wires `SkillsMiddleware` on it at `.reference/libs/deepagents/deepagents/graph.py:497-499`. Each invocation is stateless: the PM calls `task(subagent_type="document-renderer", description=<full source + requested formats>)`, receives back artifact paths plus a one-line summary in a single `ToolMessage`.

### Architect

The architect needs the broadest skill set. It must design across the full LangChain ecosystem (`langchain`), understand observability patterns (`langsmith`), design the TUI extension (`frontend-design`, `theme-factory`, `web-artifacts-builder`), author spec documents (`doc-coauthoring`), and understand model behavior from both Anthropic (`claude-api`) and OpenAI (`openai-docs`) to make informed model-specific design decisions.

**Model behavior awareness (baked into system prompt, not a skill):** The architect's system prompt must encode awareness that its primary model knowledge covers:
- Opus 4.6
- ChatGPT 5.4
- GPT 5.4 Pro

If a build is requested for a model outside this set, the architect must request the Researcher to investigate that model's behavior, prompting guidance, and provider documentation before designing for it. This is a behavioral invariant (Q12) ŌĆö it belongs in the system prompt, not as a skill.

### Planner

The planner receives the architect's design and produces a phased task document. `doc-coauthoring` is sufficient ŌĆö the planner's output is structured documentation, not code or evaluation.

### Developer

The developer needs SDK reference (`langchain`) for implementation, tracing integration (`langsmith`) for observability during development, and persistent memory (`remember`) for cross-session learning about SDK patterns and prior implementations.

### Evaluator

The evaluator validates code, SDK compliance, and UI/UX. `langsmith-evaluator-feedback` provides the feedback standard for communicating evaluation results back to the optimizer. `playwright` and `webapp-testing` enable the evaluator to spin up frontends, record interactions, and verify UI behavior ŌĆö critical for TUI acceptance testing. `remember` preserves evaluation learnings across sessions.

### Harness Engineer

The harness engineer owns evaluation science. It needs full LangSmith knowledge (`langsmith`) ŌĆö datasets, experiments, tracing, evaluation pipelines, run management. `langsmith-evaluator-feedback` provides the feedback standard for communicating harness evaluation results. `remember` preserves experiment results and calibration data across sessions.

**Agentevals SDK knowledge (baked into system prompt, not a skill):** The harness engineer must have full working knowledge of the `agentevals` SDK ŌĆö `EvaluatorResult`, LLM-as-judge, trajectory scoring, `GraphTrajectory`, tool-call matching. This is deep domain expertise that should be encoded in the system prompt as a behavioral invariant, not delegated to a skill that loads on demand. The harness engineer should *always* have this knowledge active, not progressively disclosed.

The experience the app is supposed to drive is that users can actually see data sets, experiments, etc., all being uploaded to Langsmith directly from their LLM application that is running in the TUI.

### Researcher

No skills allocated yet. The researcher's primary capability is web research and documentation retrieval. The existing `web-research` skill at `.agents/skills/web-research` is a candidate, but the design team should evaluate whether the researcher needs it or whether its research capabilities are better served by tool assignments (e.g., web search tools, documentation fetchers) rather than skills.

## Open Questions for Design Team

1. **Is the researcher's skill allocation sufficient?** Should `web-research` be added? Should the researcher get `remember` for cross-session learning about SDKs and APIs?

2. **Does the developer need more skills?** The developer is a coding agent ŌĆö should it get `frontend-design` or `web-artifacts-builder` for TUI implementation? Should it get `doc-coauthoring` for code documentation?

3. **Should the planner get `langchain`?** The planner creates development plans that reference specific SDK patterns. Having `langchain` available might improve plan accuracy.

4. **Should the harness engineer get `langchain`?** The harness engineer designs eval harnesses that test LangChain/LangGraph applications. SDK reference could improve harness design.

5. **Custom `prompt-architect` skill ŌĆö what scope?** The PM needs prompt engineering capabilities. Should this be a skill (progressive disclosure, loaded on demand) or a behavioral invariant baked into the PM's system prompt? If the PM is always crafting prompts as part of scoping, it should be in the prompt, not a skill.

6. **Skill vs. system prompt boundary.** The general principle: if the knowledge must be *always active* (the agent needs it on every call), it belongs in the system prompt as a behavioral invariant. If the knowledge is *on-demand* (the agent invokes it when needed), it belongs as a skill. The design team must evaluate each agent's skill list against this principle ŌĆö some items above might be better as prompt invariants, and some prompt invariants might be better as skills.

7. **Agentevals SDK for harness engineer ŌĆö skill or prompt?** The harness engineer needs full `agentevals` knowledge always active. This argues for system prompt inclusion, not progressive disclosure via skill. But `agentevals` is a large API surface. The design team should decide: bake the core patterns into the prompt and create an `agentevals` skill for deep reference on demand?
